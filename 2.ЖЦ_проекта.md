# 2. ЖЦ проекта

> Выбор и обоснование ЖЦ проекта. Нюансы ЖЦ.

## 2.1 Выбор жизненного цикла: Agile-модель

Для реализации проекта выбран гибкий (Agile) подход с использованием фреймворка Scrum как основного, с элементами Kanban для визуализации потока задач.

## 2.2 Обоснование выбора Agile/Scrum

| Причина выбора | Объяснение |
| :--- | :--- |
| Изменчивость требований | Требования к функционалу могут корректироваться в процессе разработки на основе обратной связи рынка или заказчика. |
| Необходимость быстрого получения обратной связи | Короткие итерации (спринты 2-3 недели) позволяют регулярно демонстрировать рабочий инкремент и вносить оперативные правки. |
| Ранний вывод ценности | Возможность выпустить минимально жизнеспособный продукт (MVP) на ранних этапах и постепенно его наращивать. |
| Повышение прозрачности и контроля | Ежедневные стендапы, бэклоги и спринт-обзоры делают прогресс и проблемы проекта видимыми для всех участников. |
| Адаптивность к изменениям | Гибкая система приоритизации в бэклоге продукта позволяет оперативно реагировать на изменения в бизнес-среде. |

## 2.3 Нюансы и адаптация выбранного ЖЦ

### 2.3.1 Гибридный подход (Scrumban)
*   Scrum будет использоваться как основа для итеративного планирования, проведения регламентных событий и выпуска версий продукта.
*   Kanban будет применен для визуализации потока задач вне спринта (например, срочные правки, техдолг), с использованием WIP-лимитов для управления загрузкой команды.

### 2.3.2 Работа в условиях ограничений
*   При фиксированных сроках/бюджете: Гибкость будет достигаться за счет управления содержанием проекта через приоритизацию бэклога. Набор функций для релиза будет определяться как "скользящая область" (scope) при фиксированных времени и стоимости.
*   С интеграцией в waterfall-процессы: Для взаимодействия с другими отделами (например, отделом закупок или тестирования на аутсорсе) будут созданы четкие интерфейсы и точки передачи артефактов в конце каждого спринта.

### 2.3.3 Критически важные факторы успеха
1.  Роль Product Owner: Необходимо выделение ответственного лица с полномочиями по формированию видения продукта, управлению бэклогом и принятию решений.
2.  Управление техническим долгом: Для предотвращения накопления "долга" в бэклог будут включаться задачи на рефакторинг, а в планы спринтов закладываться необходимый резерв.
3.  Коммуникации в распределенной команде: При географической разнесенности команды обязательны ежедневные видеостендапы, использование онлайн-инструментов (Jira, Trello, Miro) и четкое соблюдение регламентов.

## 2.4 Ключевые элементы процесса (Scrum)

| Элемент | Описание | Периодичность |
| :--- | :--- | :--- |
| Спринт | Итерация разработки с фиксированной длительностью. | 2-3 недели |
| Бэклог продукта | Упорядоченный по приоритету список всей функциональности, необходимой в продукте. | Постоянно актуализируется |
| Планирование спринта | Событие по отбору задач из бэклога продукта в бэклог спринта. | В начале каждого спринта |
| Ежедневный стендап | Короткая встреча команды для синхронизации. | Ежедневно, 15 минут |
| Обзор спринта | Демонстрация выполненной работы заинтересованным сторонам. | В конце каждого спринта |
| Ретроспектива спринта | Встреча команды для анализа процесса и улучшений. | В конце каждого спринта |

## 2.5 Риски и их минимизация

| Риск | Способ минимизации |
| :--- | :--- |
| "Размывание" требований и бесконечные изменения | Четкое определение целей спринта, наличие Definition of Ready (DoR) для задач и сильный Product Owner. |
| Накопление технического долга | Включение задач на рефакторинг и улучшение архитектуры в бэклог продукта с соответствующим приоритетом. |
| Низкое качество из-за высокой скорости | Соблюдение Definition of Done (DoD), внедрение практик автоматизированного тестирования (CI/CD). |
| Сопротивление изменениям в организации | Постепенное внедрение, обучение команды, поддержка со стороны руководства. |

## 2.6 Заключение
Выбор Agile/Scrum в качестве жизненного цикла проекта является оптимальным для условий высокой неопределенности требований и необходимости быстрой адаптации к изменениям. Учет специфических нюансов проекта, таких как возможные внешние ограничения и распределенность команды, а также грамотное управление рисками, позволят в полной мере реализовать преимущества гибкой методологии и добиться поставленных целей.
