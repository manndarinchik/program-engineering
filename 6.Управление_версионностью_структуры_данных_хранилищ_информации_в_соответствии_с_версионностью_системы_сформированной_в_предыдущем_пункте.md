# 6. Управление версионностью структурой данных хранилищ информации в соответствии с версионностью системы сформированной в предыдущем пункте

> Обеспечение устойчивости данных при миграции с одной версии на другую. Напрямую зависит от пункта 5.

## 6.1 Обзор системы управления версиями данных

В системе мониторинга ресурсов крупной IT компании должно обеспечивать:

- **Совместимость**: Новые версии приложения работают с данными старых версий
- **Откат**: Возможность вернуться к предыдущей версии системы без потери данных
- **Эволюция схемы**: Постепенное изменение структуры данных без простоев
- **Целостность данных**: Сохранение консистентности при миграциях

## 6.2 Типология изменений схемы данных

### 6.2.1 Классификация изменений

**Обратносовместимые (Forward-compatible) изменения:**
- Добавление нового столбца с NULL по умолчанию
- Добавление нового индекса
- Переименование таблицы (при условии, что старое имя остается как view)
- Можно безопасно применить без обновления приложения

**Частично совместимые изменения:**
- Удаление опционального поля
- Изменение типа данных (требует преобразования)
- Требуют согласованного обновления кода и БД

**Несовместимые (Breaking) изменения:**
- Удаление обязательного столбца
- Изменение первичного ключа
- Требуют координированного обновления
