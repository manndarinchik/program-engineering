# 1. Жизненный цикл разработки процесса

## 1.1. Выбор и обоснование жизненного цикла

Для разработки технического задания выбран гибкий подход на основе методологии Kanban. Этот выбор обусловлен спецификой задачи — созданием документации, которая требует параллельной работы над несколькими независимыми разделами и гибкого реагирования на изменения требований.

### 1.1.1. Основные преимущества выбранного подхода

* Параллельная разработка: Возможность одновременной работы нескольких исполнителей над разными частями документа без жесткой привязки к этапам водопадной модели.
* Гибкость и адаптивность: Отсутствие жестких временных рамок спринтов позволяет свободно вносить изменения, добавлять новые задачи и пересматривать приоритеты по мере необходимости.
* Визуализация потока работ: Использование Kanban-досок обеспечивает прозрачность процесса для всех участников, позволяет легко отслеживать статус каждой задачи и выявлять узкие места.
* Фокус на непрерывном потоке: Цель — обеспечить плавное и непрерывное продвижение задач от создания до завершения, минимизируя простои и перегрузки исполнителей.
* Снижение бюрократических накладных расходов: Простота методологии позволяет сосредоточиться на содержательной работе, а не на соблюдении формальных процедур.

## 1.2. Принципы работы над документом

### 1.2.1. Организация рабочего процесса

Процесс организован вокруг трех ключевых эпиков (тематических блоков), каждый из которых содержит набор взаимосвязанных задач. Управление ведется с помощью виртуальных Kanban-досок со следующими стандартными колонками:

1. Open (Открыто): Задачи сформулированы, готовы к взятию в работу.
2. In progress (В работе): Задачи, над которыми ведется активная работа.
3. Review (На проверке): Завершенные задачи, ожидающие проверки и согласования.
4. Done (Выполнено): Принятые и завершенные задачи.

### 1.2.2. Правила работы с задачами

1. Взятие задач: Задачи из колонки "In progress" берутся разработчиками самостоятельно, исходя из их загруженности и приоритета задачи.
2. Лимиты WIP (Work In Progress): Для каждого разработчика устанавливается лимит одновременно выполняемых задач (не более 2-3). Это предотвращает перегрузку и способствует завершению начатых работ.
3. Процесс проверки: Завершение задачи подразумевает ее перемещение в "Review", где она проверяется другими участниками команды или менеджером потока на соответствие требованиям.
4. Критерии завершения: Задача считается выполненной только после:
   - Проверки качества контента
   - Проверки соответствия формату документа
   - Утверждения ответственным за эпик
5. Ежедневные стендапы: Проводятся короткие (10-15 минут) ежедневные встречи для синхронизации:
   - Что сделано с последнего стендапа
   - Что планируется сделать до следующего
   - Какие есть блокировки и проблемы

## 1.4. Цели и ожидаемые результаты

### 1.4.1. Ключевые цели документа
Данный документ (техническое задание на проект) должен выполнить следующие ключевые цели:

1. Создать единую и непротиворечивую основу для всего последующего процесса разработки программного продукта.
2. Четко определить и формализовать все аспекты управления проектом, включая жизненные циклы, артефакты и процессы выпуска.
3. Служить источником истины для всех участников проекта (заказчиков, разработчиков, тестировщиков, менеджеров) на начальном этапе.
4. Обеспечить управляемость и предсказуемость проекта через установление регламентов и процедур.
5. Минимизировать риски недопонимания и технических долгов за счет детального описания требований к процессам до начала активной разработки.

### 1.4.2. Критерии успеха

- Все эпики завершены в полном объеме
- Документ прошел внутреннее ревью всех участников команды
- Содержание документа соответствует исходным требованиям
- Формат и структура документа единообразны
- Сроки разработки соблюдены
