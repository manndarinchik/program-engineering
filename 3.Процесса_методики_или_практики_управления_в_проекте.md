# 3. Процессы, методики и практики управления в проекте

> Выбор и обоснование структуры управления проекта. Ролевой состав команды проекта, способ и принцип управления, как происходит контроль прогресса и приемка готового продукта. Каждый процесс должен иметь документальное сопровождение с определенными стадиями с указанием ответственных.

## 3.1 Структура управления проектом

В соответствии с выбранной Agile-моделью (Scrum), применяется гибкая структура управления с упором на самоорганизующуюся команду и эмпирический контроль. Управление фокусируется не на соблюдении изначального плана, а на обеспечении потока ценности, прозрачности процесса и быстрой адаптации к изменениям.

Принципы управления:
*   Эмпиризм: Управление основывается на наблюдении, опыте и адаптации (прозрачность, инспекция, адаптация).
*   Самоорганизация: Команда сама определяет наилучший способ выполнения работы в рамках целей спринта.
*   Ориентация на ценность: Приоритеты определяются ценностью для бизнеса и пользователя.
*   Итеративность и инкрементальность: Прогресс достигается через последовательность законченных рабочих версий продукта.

## 3.2 Ролевой состав команды проекта

| Роль | Обязанности и ответственность | Ключевые артефакты / События |
| :--- | :--- | :--- |
| Владелец продукта | Формирует видение продукта, управляет бэклогом продукта (создание, приоритизация, уточнение),  Главное "лицо" бизнеса в команде. | Бэклог продукта, Спринт-обзор, Планирование спринта. |
| Архитектор продукта | Принимает результаты работы (инкремент) в конце спринта. Технический лидер продукта. | Спринт-обзор, Планирование спринта. |
| Scrum-мастер | Отвечает за понимание и соблюдение командой принципов и правил Scrum. Устраняет организационные препятствия (impediments), фасилитирует события, помогает команде повышать эффективность. | Все Scrum-события, Доска задач, Ретроспектива. |
| Разработчики | Кросс-функциональная, самоорганизующаяся группа (3-9 человек). Включает: Backend/Frontend-разработчиков, тестировщиков, DevOps-инженеров и др. Совместно несут ответственность за создание рабочего инкремента в конце каждого спринта. | Бэклог спринта, Ежедневный стендап, Инкремент продукта. |
| Стейкхолдеры | Предоставляют обратную связь, участвуют в демонстрациях, формулируют потребности. Не входят в ежедневную работу команды, но критически важны для обратной связи. | Спринт-обзор. |

## 3.3 Приемка готового продукта (Definition of Done, DoD)

Приемка осуществляется итерационно, в конце каждого спринта, на основе единого для всей команды "Определения Готовности" (Definition of Done).
Процесс приемки инкремента (в конце спринта):
1.  Демонстрация: Команда представляет рабочий инкремент стейкхолдерам на Спринт-обзоре.
2.  Проверка: Product Owner проверяет выполнение всех критериев DoD и соответствие функциональности описанию в бэклоге продукта.
3.  Принятие решения: Product Owner принимает или отклоняет результаты спринта.
    *   Принято: Инкремент считается готовым и может быть выпущен (если это предусмотрено планом релиза). Задачи закрываются.
    *   Не принято: Выявленные недоработки вносятся в бэклог продукта как новые элементы и приоритезируются для следующих спринтов.

Пример Definition of Done (DoD):
*   Код написан, проверен (code review) и слит в основную ветку (main branch).
*   Все автоматизированные тесты (unit, integration) пройдены успешно.
*   Проведено ручное тестирование (если требуется) и все критические баги исправлены.
*   Код соответствует стандартам и руководствам по стилю проекта.
*   Документация (комментарии в коде, README) обновлена.
*   Функциональность развернута на тестовом/стейджинг-окружении и протестирована в нем.

## 3.4 Ключевые документы и артефакты процесса

| Артефакт / Документ | Назначение | Ответственный за актуализацию | Связь с событиями |
| :--- | :--- | :--- | :--- |
| Дорожная карта продукта (Product Roadmap) | Визуализация стратегического видения и плана выпуска основных функций на 6-12 месяцев вперед. | Product Owner | Основа для планирования релизов. |
| Бэклог продукта (Product Backlog) | Живой, приоритизированный список всех требований, улучшений и исправлений. Основной источник работ для команды. | Product Owner (с участием команды) | Основной артефакт для Планирования спринта и Спринт-обзора. |
| Бэклог спринта (Sprint Backlog) | Набор элементов из бэклога продукта, отобранных для реализации в текущем спринте + план их выполнения от команды. | Команда разработчиков | Создается на Планировании спринта, актуализируется ежедневно. |
| Инкремент продукта | Сумма всех завершенных элементов бэклога продукта на конец спринта. Измеримый результат работы. | Команда разработчиков | Демонстрируется на Спринт-обзоре. |
| Доска задач (Kanban/Scrum Board) | Визуальное отображение статуса задач в рамках спринта (To Do, In Progress, Code Review, Testing, Done). | Команда разработчиков | Используется на Daily Scrum для синхронизации. |
| Протокол ретроспективы | Документ с выводами и планом конкретных улучшений на следующий спринт по итогам ретроспективы. | Scrum-мастер (фасилитатор) | Создается на Ретроспективе спринта. |
